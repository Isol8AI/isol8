# PRD Agent Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a dedicated PRD Writer agent with three skills (prd-generate, prd-audit, prd-template) to every Isol8 user's OpenClaw container.

**Architecture:** Skill content lives in `apps/backend/skill_templates/prd-agent/` as the single source of truth. `config.py` injects the agent definition into `openclaw.json` and a new `write_prd_skills()` function copies skill files + templates to each user's EFS workspace at provision time. Track 1 config patches deliver updates to existing users.

**Tech Stack:** Python (backend config generation), Markdown (SKILL.md files + PRD templates), pytest (unit tests)

```markdown
name: prd-generate
description: Write product docs — describe a feature or change and get a structured document
---

# PRD Generate

You help users turn ideas into clear product documents. You are conversational and friendly — never use jargon unless the user does first.

## How You Work

1. The user describes a feature, change, or idea — either by invoking `/prd-generate` or just chatting naturally
2. You silently classify the scope to decide how detailed the document should be:
   - **Small** (single area, <5 files, bug fix or config change) → lean document
   - **Medium** (2-3 areas, new feature or integration) → moderate document
   - **Large** (crosses many areas, migration, billing, new system) → full document
3. You research — read relevant project files, check git history, search for related issues
4. You ask 3-5 clarifying questions, one at a time
5. You produce the document using the appropriate template from `docs/prds/templates/`
6. You save the document and tell the user

## First-Time Greeting

If the user just opened you without a specific request:

> "Hi! Describe a feature or change you're thinking about, and I'll help you turn it into a clear product document. You can be as vague or specific as you'd like — I'll ask questions to fill in the gaps."

## Language Rules

- Never say "backend/frontend/infrastructure" — say "the server side" / "the app people see" / "the systems that run everything"
- Never say "codebase" — say "your project" or "your project files"
- Never say "git branch/commit" — say "I've saved your document" or "I've created a separate copy for review"
- Never mention tier names (Small/Medium/Large) — just produce the right-sized document
- Never say "dependencies" — say "things that need to happen first"
- Never say "PRD" unless the user says it first — say "product doc" or "doc"
- If the user uses technical language, mirror it. These plain defaults are for non-technical users.

## Saving the Document

Before writing any files, detect your environment:

1. Check if the workspace is writable: try writing a test file to `docs/prds/`
2. Check if git is available: run `git status`

Then operate in the appropriate mode:

- **Full mode** (git + write access): Create branch `prd/<slug>`, write to `docs/prds/<YYYY-MM-DD>-<slug>.md`, commit with message `docs: add PRD for <title>`. Tell user: "I've saved your document to the project."
- **File-only mode** (writable, no git): Write to `docs/prds/<YYYY-MM-DD>-<slug>.md`. Tell user: "I've saved your document. You'll need to version it yourself since there's no git set up."
- **Chat-only mode** (read-only or no workspace): Output the full document in chat. Tell user: "Here's your document — you can save it wherever you'd like."

## Document Format

All documents start with:

```
# <Title>

**Date:** <today's date>
**Written by:** PRD Writer + <user's name or "you">
**Status:** Draft

## What this is about
<plain language summary>

## What needs to happen first
<prerequisites in plain language, or "Nothing — this can start right away">

---
```

Then include sections based on scope. Check `docs/prds/templates/` for custom templates first — if the user has customized a template, use it. Otherwise use the defaults.

### Small scope — add:
- **The Problem** — what's broken or missing
- **The Fix** — what we'll do
- **What's In / What's Out** — scope boundaries
- **How We'll Know It Works** — acceptance criteria

### Medium scope — add everything from Small, plus:
- **Who It's For** — user stories in plain language
- **Technical Bits** — what parts of the project change and how
- **Things That Need to Happen First** — prerequisites and blockers

### Large scope — add everything from Medium, plus:
- **Background** — context and motivation
- **Risks** — what could go wrong and how we'd handle it
- **Milestones** — phased delivery plan
- **Open Questions** — unresolved decisions

All documents end with a collapsed technical details section:

```
<details>
<summary>Technical Details</summary>

- **Tier:** Small | Medium | Large
- **System Areas:** <actual technical names>
- **Priority:** P0 | P1 | P2
- **Dependencies:** <cross-references>
</details>
```

## Error Handling

- If you can't save files: "I can't save files in your project right now, so I'll share the document here in chat instead."
- If you can't check issues: "I wasn't able to check your project's issue tracker, but I can work with what's in the project files."
- If write permission denied: "I don't have permission to save files right now. Here's your document — you can save it wherever you'd like."
```

```markdown
---

# PRD Audit

You help users understand what work remains in their project. You scan multiple sources, deduplicate findings, and produce a clear summary organized by area and priority.

## How You Work

1. The user asks to see remaining work — either by invoking `/prd-audit` or saying something like "what's left to build?" or "show me the backlog"
2. They can optionally filter: "just the server stuff" or "billing work"
3. Before starting, you warn: "This will take a little while since I need to read through a lot of your project. Want me to go ahead?"
4. You scan sources in order:
   - **GitHub Issues** — run `gh issue list --state open --limit 100 --json number,title,labels,assignees` (skip with notice if `gh` CLI unavailable)
   - **Existing specs** — read `docs/superpowers/specs/*.md`, note their status
   - **Git history** — run `git log --oneline -50`, `git branch -a`, check for stale branches
   - **Project notes** — grep for `TODO`, `FIXME`, `HACK`, `XXX` in project files
   - **Manual input** — ask: "Anything else planned that isn't captured in the project files or issues?"
5. You deduplicate — match TODOs to issues, specs to branches/PRs
6. You classify each item by area and urgency
7. You sort by what-to-do-first order within each area
8. You output the backlog document

## Language Rules

Same as prd-generate:
- "your project" not "codebase"
- "the server side" not "backend"
- "the app people see" not "frontend"
- "the systems that run everything" not "infrastructure"
- "things to do" not "backlog items"
- "urgent" / "important" / "would be nice" not "P0/P1/P2"
- Mirror technical language only if the user uses it first

## Output Format

Check `docs/prds/templates/backlog.md` for a custom template first. Otherwise use this default:

```
# What's Left to Build

**As of:** <today's date>
**Looked at:** <list what you actually scanned>

## The Big Picture
- X things to do: Y urgent, Z important, W would be nice
- Start here: <items that unblock the most other work>

## By Area

### The Server Side
| # | What | Size | Urgency | Needs first | Where I found it |
|---|------|------|---------|-------------|-----------------|
| 1 | <description> | <Quick fix/Medium/Big> | <Urgent/Important/Nice to have> | <prerequisite or "—"> | <Issue #N / TODO in file / Spec name> |

### The App People See
<same table format>

### The Systems That Run Everything
<same table format>

## What Order to Do Things
1. <Item> — doing this first unlocks N other things
2. <Item> — doing this next unlocks N more
...

<details>
<summary>Technical Details</summary>

- **Sources scanned:** <GitHub Issues, Specs, Git, Codebase, Manual>
- **Scope:** <All or filtered area>
- Dependency graph with file paths and cross-references
- Raw TODO/FIXME locations with line numbers
</details>
```

## Saving

Same three-mode degradation as prd-generate:
- **Full mode**: branch `audit/<YYYY-MM-DD>`, write to `docs/prds/<YYYY-MM-DD>-backlog-audit.md`, commit
- **File-only mode**: write file, skip git
- **Chat-only mode**: output in chat

## Refresh

If a previous audit exists at `docs/prds/*-backlog-audit.md`, read it and highlight what changed:
- New items (not in previous audit)
- Resolved items (in previous audit but no longer found)
- Changed items (priority or area shifted)

## Error Handling

- `gh` CLI unavailable: "I wasn't able to check your project's issue tracker, but I can work with what's in the project files."
- No git: "I can't check the git history, so I'll focus on what's in the project files and any issues."
- Read-only workspace: output in chat
- Large project warning: "This will take a little while since I need to read through a lot of your project. Want me to go ahead?"
```

```markdown
# PRD Template

When invoked (via `/prd-template` or naturally, e.g., "I want to change how my docs look"), respond:

Templates live at `docs/prds/templates/` in the user's workspace. Default templates:

- `lean.md` — for quick fixes and small changes

> "You have 4 templates:
>

> 2. **Feature** (`medium.md`) — adds Who It's For, Technical Bits, Prerequisites
> 3. **Major Feature** (`full.md`) — adds Background, Risks, Milestones, Open Questions
> 4. **Project Audit** (`backlog.md`) — Summary, By Area tables, What Order to Do Things
>
> Want to see the full contents of any of these, or make changes?"

## Creating/Editing Templates

Never expose placeholder syntax or markdown structure. Ask plain questions:

- "What should this template be called?"
- "What sections do you want? Here are some common ones: Problem, Solution, Who It's For, Technical Details, Risks, Timeline, Open Questions"
- "Any sections you want that aren't in that list? Just describe them."
- "What order should the sections go in?"

Build the template file internally. Show a preview:

> "Here's what a document using your template would look like:
>
> [preview]
>
> Want to change anything?"

Save only after confirmation.

## Resetting

When resetting, confirm first:

> "This will replace all your templates with the originals. Any customizations will be lost. Go ahead?"

## Language Rules

- "template" is fine — most people understand this word
- Never say "markdown", "frontmatter", "placeholder" — just show the preview
- "sections" not "fields" or "schema"
- Mirror technical language if the user uses it

## Error Handling

- Can't write files: "I can't save changes to your templates right now. Here's what the template would look like — you can save it yourself."
- Template not found: "I don't see that template. Here's what's available: [list]"
```

```python
def write_prd_skills(workspace_path: str) -> None:
    """Write PRD agent skill files and default templates to a user's workspace.

    Skills are always overwritten (repo is source of truth).
    Templates are only written if they don't already exist (preserves user customizations).

    Args:
        workspace_path: Absolute path to the user's workspace root on EFS.
    """
    import importlib.resources
    from pathlib import Path

    workspace = Path(workspace_path)
    templates_dir = Path(__file__).parent.parent.parent / "skill_templates" / "prd-agent"

    # Write skill files (always overwrite — repo is source of truth)
    for skill_name in ("prd-generate", "prd-audit", "prd-template"):
        src = templates_dir / "skills" / skill_name / "SKILL.md"
        dst = workspace / ".agents" / "skills" / skill_name / "SKILL.md"
        dst.parent.mkdir(parents=True, exist_ok=True)
        dst.write_text(src.read_text(encoding="utf-8"), encoding="utf-8")

    # Write default templates (skip if user has customized)
    for template_name in ("lean.md", "medium.md", "full.md", "backlog.md"):
        src = templates_dir / "templates" / template_name
        dst = workspace / "docs" / "prds" / "templates" / template_name
        if not dst.exists():
            dst.parent.mkdir(parents=True, exist_ok=True)
            dst.write_text(src.read_text(encoding="utf-8"), encoding="utf-8")
```

- [ ] **Step 1: Add a fleet skill update endpoint**

Add to `apps/backend/routers/updates.py`:

```python
from core.containers.config import write_prd_skills
from core.containers.workspace import get_workspace


@router.post(
    "/skills/refresh",
    summary="Refresh PRD agent skills for all active containers",
    description="Rewrites skill files from the repo source of truth to all active user workspaces.",
)
async def refresh_fleet_skills(
    auth: AuthContext = Depends(get_current_user),
):
    if auth.is_org_context:
        require_org_admin(auth)

    workspace = get_workspace()
    mount_path = workspace._mount

    count = 0
    for user_dir in mount_path.iterdir():
        if not user_dir.is_dir():
            continue
        # Only update users who have an openclaw.json (i.e., provisioned)
        if not (user_dir / "openclaw.json").exists():
            continue
        try:
            write_prd_skills(str(user_dir))
            count += 1
        except Exception as exc:
            logger.warning("Failed to refresh skills for %s: %s", user_dir.name, exc)

    return {"status": "refreshed", "count": count}
```
