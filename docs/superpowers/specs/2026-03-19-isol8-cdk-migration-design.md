# Isol8 CDK Migration — Design Spec

**Date:** 2026-03-19
**Status:** Approved

## Overview

Migrate isol8's infrastructure from Terraform (HCL) to AWS CDK (TypeScript). Same architecture, same compute model (EC2 ASG), new database (RDS PostgreSQL replacing Supabase). The CDK project lives at `apps/infra/` in the isol8 repo, following the same `lib/app.ts` + `lib/stacks/` pattern as goosetown.

## Motivation

- Consistent IaC across both products (isol8 + goosetown both on CDK)
- CDK provides first-class constructs for API Gateway, ECS, and RDS
- Eliminates terraform state management overhead
- Enables productionization in the final architecture (no throwaway terraform prod work)
- Replaces external Supabase dependency with AWS-managed RDS

## Architecture

No architectural changes — same compute model, same networking, same API Gateway pattern.

```
Vercel (Frontend)
  ↓ HTTPS
HTTP API Gateway → VPC Link v2 → ALB (internal) → EC2 ASG (FastAPI)
                                                      ↓ boto3
WebSocket API GW → VPC Link v1 → NLB → ALB → EC2    ECS Fargate (per-user OpenClaw)
                                                      ↕
                                                    EFS (agent workspaces)
                                                    RDS PostgreSQL (replaces Supabase)
                                                    Cloud Map (service discovery)
```

## CDK Structure

```
apps/infra/
  lib/
    app.ts                      ← CDK entry point + stack orchestration
    stacks/
      network-stack.ts          ← VPC, subnets, NAT
      database-stack.ts         ← RDS PostgreSQL
      auth-stack.ts             ← Secrets Manager, KMS
      dns-stack.ts              ← ACM certs, Route53
      compute-stack.ts          ← EC2 ASG, ALB, ECR, IAM
      container-stack.ts        ← ECS Fargate cluster, Cloud Map, EFS
      api-stack.ts              ← HTTP + WebSocket API Gateways, VPC Links, NLB, Lambda authorizer
  cdk.json
  package.json
  tsconfig.json
```

## Stack Details

### 1. NetworkStack

- VPC with 2 AZs (dev) / 3 AZs (prod)
- Public subnets with NAT gateway
- Private subnets for compute, database, containers
- Default security group restrictions

**Exports:** `vpc`

### 2. DatabaseStack

- RDS PostgreSQL instance
  - `db.t3.small` (dev) / `db.t3.medium` (prod)
  - Private subnet placement
  - Multi-AZ: false (dev) / true (prod)
  - Automated backups, encryption at rest via KMS
  - Security group: only EC2 ASG and ECS tasks can connect on port 5432
- Database credentials stored in Secrets Manager (auto-generated by RDS construct)

**Exports:** `dbInstance`, `dbSecurityGroup`, `dbSecret` (connection credentials)

### 3. AuthStack

- Secrets Manager secrets (values populated out-of-band, not in CDK):
  - `isol8/{env}/clerk_secret_key`
  - `isol8/{env}/clerk_webhook_secret`
  - `isol8/{env}/stripe_secret_key`
  - `isol8/{env}/stripe_webhook_secret`
  - `isol8/{env}/huggingface_token`
  - `isol8/{env}/perplexity_api_key`
  - `isol8/{env}/encryption_key`
- KMS key for EBS volume encryption, EFS encryption, and general-purpose encryption

**Exports:** `secrets` (map of all secret references), `kmsKey`

### 4. DnsStack

- ACM certificate for `*.isol8.co` (wildcard) with DNS validation
- Route53 hosted zone lookup for `isol8.co`
- No A records created here (other stacks create their own to avoid circular dependencies)

**Exports:** `certificate`, `hostedZone`

### 5. ComputeStack

- ECR repository for backend Docker images
- EC2 Auto Scaling Group:
  - Launch template with user data script
  - Instance type: `t3.large` (downsized from `r5.xlarge` after town code removal)
  - AMI: Amazon Linux 2023
  - User data: install Docker, pull image from ECR, configure env vars from Secrets Manager, run FastAPI
  - EFS mount for agent workspaces
  - Min/max: 1/2 (dev), 1/3 (prod)
- ALB (internal):
  - HTTPS listener with ACM cert
  - 300s idle timeout (for SSE streaming)
  - Health check on `/health`
  - Sticky sessions enabled
- IAM instance profile with permissions for:
  - ECR pull, ECS task management, EFS mount
  - Secrets Manager read, Bedrock invoke
  - CloudWatch logs, KMS decrypt
  - Cloud Map service discovery
- No DNS records here — ApiStack creates custom domain records for `api-{env}.isol8.co` and `ws-{env}.isol8.co`

**Exports:** `alb`, `asg`, `repository`, `instanceRole`, `ec2SecurityGroup`

### 6. ContainerStack

- ECS Fargate cluster in private subnets
- AWS Cloud Map namespace (`isol8-{env}.local`) for service discovery
- EFS filesystem:
  - Encrypted via KMS
  - Mount targets in private subnets
  - Access points for per-user isolation
  - Security group: EC2 and Fargate tasks can mount
- ECS task execution role (ECR pull, CloudWatch logs)
- ECS task role (Bedrock invoke, EFS mount, S3 access if needed)

**Exports:** `cluster`, `cloudMapNamespace`, `efsFileSystem`, `taskExecutionRole`

### 7. ApiStack

- HTTP API Gateway (APIGatewayV2):
  - Custom domain: `api-{env}.isol8.co` with ACM cert
  - Default route → VPC Link v2 → ALB
  - Rate limiting
  - Route53 A record for custom domain
- WebSocket API Gateway:
  - Custom domain: `ws-{env}.isol8.co` with ACM cert
  - `$connect` route → Lambda authorizer (Clerk JWT validation)
  - `$disconnect` route → Lambda (cleanup)
  - `$default` route → VPC Link v1 → NLB → ALB → EC2
  - Route53 CNAME record for custom domain
- NLB in private subnets (required for WebSocket API GW VPC Link v1)
- Lambda authorizer function (Python, validates Clerk JWT via JWKS)
- Management API endpoint output (for EC2 to push WebSocket messages)

**Exports:** `httpApiUrl`, `webSocketUrl`, `managementApiUrl`

### app.ts — Stack Orchestration

```
NetworkStack          (no deps)
DatabaseStack         (depends on: network, auth)
AuthStack             (no deps)
DnsStack              (no deps)
ComputeStack          (depends on: network, database, auth, dns, container)
ContainerStack        (depends on: network, auth)
ApiStack              (depends on: network, compute, auth, dns)
```

GitHub Actions OIDC role created inline in `app.ts` with permissions for: CDK deploy (CloudFormation, S3, IAM), ECR push, ECS/ASG management, Secrets Manager read.
CDK Nag (`AwsSolutionsChecks`) applied to all stacks.
Tags: `Project=isol8`, `Environment={env}` on all resources.

## CI/CD

Two-stage model (dev + prod):

```
PR opened    → CDK diff posted as PR comment
             → Backend tests run (no deploy)
             → Vercel preview deployment (env vars point to dev backend)

Merge to main → CDK deploy dev (automatic)
             → Backend build Docker + deploy dev (automatic)
             → CDK deploy prod (manual approval via 'prod' GitHub environment)
             → Backend deploy prod (manual approval)
             → Vercel production deployment (automatic, env vars point to prod backend)
```

Workflows:
- `infra.yml` — CDK diff on PR, CDK deploy on merge (dev auto, prod gated)
- `backend.yml` — test on PR, build+deploy on merge (dev auto, prod gated)
- Frontend — Vercel auto-deploy (Preview on PR, Production on merge)

## Migration Strategy

### Phase 1: Create CDK Project
- Initialize CDK in `apps/infra/`
- Write all 7 stacks
- Synth and verify

### Phase 2: Deploy CDK Dev (Alongside Terraform)
- Bootstrap CDK in the AWS account (already done for goosetown)
- Deploy CDK stacks to dev
- Create RDS instance, run database schema migration
- Update EC2 user data to use RDS connection string instead of Supabase
- Verify backend works with CDK-managed infra

### Phase 3: Cut Over
- Update DNS records from terraform-managed to CDK-managed
- Update CI/CD workflows (replace terraform with CDK)
- Verify all endpoints work

### Phase 4: Tear Down Terraform
- `terraform destroy` for dev environment
- Remove `apps/terraform/` directory
- Remove terraform workflow

### Phase 5: Deploy Prod
- CDK deploy prod (`-c env=prod`)
- Set up prod secrets
- Set up Vercel prod domain + env vars
- DNS cutover for prod domains
- End-to-end verification

## What Changes in Backend Code

Minimal:
- `DATABASE_URL` env var changes from Supabase connection string to RDS connection string
- Both are PostgreSQL — SQLAlchemy works identically
- No code changes required, only environment variable update

## What's NOT Changing

- Backend application code (FastAPI, SQLAlchemy, all routes/services)
- Frontend (Vercel, same deployment model)
- Clerk auth (same project, same keys)
- ECS Fargate container orchestration pattern
- Docker build/push pipeline
- WebSocket architecture (API GW → VPC Link → NLB → ALB → EC2)

## Relationship to GooseTown

Isol8 and GooseTown are **completely separate** CDK projects:
- GooseTown: `Isol8AI/goosetown/apps/infra/` — Fargate, DynamoDB, 7 stacks
- Isol8: `Isol8AI/isol8/apps/infra/` — EC2 ASG, RDS PostgreSQL, 7 stacks

They share:
- Same AWS account (877352799272)
- Same Clerk auth project
- Same Route53 hosted zone (isol8.co)
- Same CDK bootstrap (already done)

They do NOT share: VPCs, databases, compute, secrets, CI/CD pipelines, or CDK stacks.
